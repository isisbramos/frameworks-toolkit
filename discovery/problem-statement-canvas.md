# Problem Statement Canvas

## When to use
When aligning the team on what problem you're actually solving — before anyone starts designing or building.

## Template

### 1. Who is affected?
> [Describe the user or segment]
>
> ### 2. What is the problem?
> > [Describe the pain point or unmet need]
> >
> > ### 3. Where does it happen?
> > > [Context — in what situation, workflow, or environment]
> > >
> > > ### 4. When does it happen?
> > > > [Frequency, triggers, timing]
> > > >
> > > > ### 5. Why does it matter?
> > > > > [Impact on the user and/or the business]
> > > > >
> > > > > ### 6. How do we know this is real?
> > > > > > [Evidence — user research, data, support tickets, etc.]
> > > > > >
> > > > > > ## Output
> > > > > > A clear, one-sentence problem statement:
> > > > > >
> > > > > > > **[Who]** experiences **[what problem]** when **[context]**, which results in **[impact]**.
> > > > > > >
> > > > > > > ## Tips
> > > > > > >
> > > > > > > - Write it before the solution conversation starts
> > > > > > > - - Revisit it when scope creep happens
> > > > > > >   - - If you can't fill "How do we know this is real?" — you need more discovery
