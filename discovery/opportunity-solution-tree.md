# Opportunity Solution Tree

## When to use
When you need to map opportunities before jumping to solutions. Helps prevent the team from falling in love with a solution before understanding the problem space.

## Structure

```
Desired Outcome
├── Opportunity 1
│   ├── Solution A
│   │   └── Experiment 1
│   └── Solution B
│       └── Experiment 2
├── Opportunity 2
│   └── Solution C
│       └── Experiment 3
└── Opportunity 3
    └── Solution D
        └── Experiment 4
```

## How to fill it

1. **Start with the desired outcome** — what business or product result are you trying to achieve?
2. 2. **Map opportunities** — what user needs, pain points, or desires could drive that outcome?
   3. 3. **Generate solutions** — for each opportunity, brainstorm possible solutions
      4. 4. **Design experiments** — for each solution, define how you'd test it
        
         5. ## Tips
        
         6. - Keep opportunities framed as user needs, not features
            - - One outcome per tree
              - - Prioritize opportunities before generating solutions
                - - This is a living document — update it as you learn
                 
                  - ---

                  *Based on Teresa Torres' Opportunity Solution Tree framework.*
